# C# Bindings Documentation Plan

## Overview

NavioBlsct is a .NET P/Invoke binding for the libblsct C library. This document describes the implementation of XML documentation comments and DocFX-based documentation generation for the C# bindings.

## XML Documentation Comments

All public and internal members in `ffi/csharp/Blsct.cs` are documented with XML doc comments (`///`). These comments are:

1. **Embedded in source code** ‚Äî live alongside the implementation
2. **Extracted into an XML file** by `<GenerateDocumentationFile>true</GenerateDocumentationFile>` in `NavioBlsct.csproj`
3. **Packaged with the NuGet** ‚Äî `NavioBlsct.xml` ships alongside `NavioBlsct.dll` in the NuGet package
4. **Shown as IntelliSense** in Visual Studio when consumers reference the package

### Documentation Coverage

#### Enum `AddressEncoding`
- `Bech32 = 0` ‚Äî Legacy Bech32 encoding (RFC-compatible, original Bitcoin variant)
- `Bech32M = 1` ‚Äî Bech32m encoding (BIP-350, recommended)

#### Class `Blsct` (static)
All public static methods:

| Method | Wraps native | Documentation |
|--------|---|---|
| `GenSubAddrId(long account, ulong address)` | `gen_sub_addr_id` | Returns opaque handle; caller must free with `FreeObj` |
| `DeriveSubAddress(byte[] viewKey, byte[] spendKey, IntPtr subAddrId)` | `derive_sub_address` | Validates key sizes (32 and 48 bytes); throws `ArgumentException`/`ArgumentNullException` |
| `EncodeAddress(IntPtr subAddr, AddressEncoding encoding = AddressEncoding.Bech32M)` | `encode_address` | Returns managed string; frees native resources internally; throws `InvalidOperationException` on failure |
| `DecodeAddress(string blsctEncAddr)` | `decode_address` | Parses address string; throws `ArgumentException` on blank input, `InvalidOperationException` on native failure; returns handle (caller must free) |
| `FreeObj(IntPtr handle)` | `free_obj` | Frees native handle; no-op on `IntPtr.Zero` |

#### Internal Helpers
Used in unit tests (via `InternalsVisibleTo`):

| Method | Purpose |
|--------|---------|
| `EnsureSuccess(IntPtr retVal)` | Reads result code; throws `InvalidOperationException` if non-zero |
| `ReadResultCode(IntPtr retVal)` | Reads first byte (result code) from native `RetVal` struct |
| `ReadValuePtr(IntPtr retVal)` | Reads pointer at offset `IntPtr.Size` from native `RetVal` struct |

All three include remarks describing the native `RetVal` struct layout.

### Native RetVal Structure

Native functions return a pointer to a struct:

```
[0]             byte    result code (0 = success, non-zero = error)
[IntPtr.Size]   IntPtr  value pointer (actual payload)
[IntPtr.Size*2] nuint   value size
```

This layout is documented in class-level remarks and individual helper remarks.

1. XML documentation comments extracted from `Blsct.cs`
2. Hand-written Markdown articles in `ffi/csharp/docs/articles/`

### Key Configuration (docfx.json)

- **Metadata source:** `../NavioBlsct.csproj` ‚Äî DocFX extracts XML comments from the compiled assembly
- **Metadata output:** `api/` ‚Äî generated YAML files (ignored by Git)
- **Include private members:** `false` ‚Äî private `DllImport` declarations hidden from docs
- **Template:** `default` ‚Äî standard DocFX styling
- **Search:** enabled

### Articles

Hand-written documentation:

1. **installation.md** ‚Äî How to install the NuGet package, system requirements
2. **memory-management.md** ‚Äî Rule: always call `FreeObj` on returned handles; safe no-op patterns; `EncodeAddress` self-managing behavior
3. **return-value-layout.md** ‚Äî Deep dive into native `RetVal` struct, offset table, internal helper methods

## Building Documentation

### Prerequisites

```bash
dotnet tool install -g docfx
```

### Generate and View

```bash
cd ffi/csharp/docs
docfx docfx.json        # Generate HTML to _site/
docfx docfx.json --serve  # Preview at http://localhost:8080
```

### Build Output

- `_site/` ‚Äî generated HTML documentation (git-ignored)
- `api/*.yml` ‚Äî auto-generated YAML metadata (git-ignored)
- `api/index.md` ‚Äî auto-generated API index (git-ignored)

## CI/CD Integration

### Workflow: csharp-publish-doc.yml

Automatic deployment to `gh-pages` branch:

**Trigger:** Push to `main` when files under `ffi/csharp/Blsct.cs` or `ffi/csharp/docs/**` change

**Steps:**
1. Checkout code
2. Set up .NET 8
3. Install DocFX CLI globally
4. Run `docfx docfx.json` in `ffi/csharp/docs/`
5. Deploy `_site/` to `gh-pages` branch in `csharp/` subdirectory

**Result:** Documentation published at `https://user.github.io/repo/csharp/` alongside `/python/` and `/ts/` docs

**Preservation:** `keep_files: true` ensures existing `/python` and `/ts` directories are not deleted

## NuGet Package Integration

When building the NuGet package:
```bash
cd ffi/csharp
dotnet pack NavioBlsct.csproj --configuration Release
```

The `NavioBlsct.xml` file is automatically included because `<GenerateDocumentationFile>true</GenerateDocumentationFile>` is set in `NavioBlsct.csproj`.

## Consistency with Other Bindings

- **Python:** Uses Sphinx + reStructuredText; docs at `/python/` on gh-pages
- **TypeScript:** Uses TypeDoc; docs at `/ts/` on gh-pages
- **C#:** Uses DocFX + Markdown; docs at `/csharp/` on gh-pages
